# شرح مشروع Threat Hunters بالعربي

## 1) فكرة المشروع
المشروع عبارة عن منصة Security Intelligence بتعرض:

- فحص للويب والمواقع
- فحص للباسورد
- فحص للإيميل ضد بيانات التسريب
- تقارير PDF تفصيلية
- داشبورد فيه ملخصات، نتائج، وتاريخ الفحوصات

الهدف الأساسي إن المستخدم ياخد نتيجة واضحة ومفهومة، مش مجرد أرقام أو كلام عام.

## 2) ليه استخدمنا التقسيمة دي

### Frontend
استخدمنا `React + Vite` لأن:

- سريع في التطوير
- مناسب للواجهات التفاعلية
- سهل نفصل الصفحة إلى Components واضحة
- مناسب لعرض بيانات كثيرة بشكل منظم

### Backend
استخدمنا باك إند منفصل لأن:

- فحص التسريبات والبريد لازم يفضلوا Server-side
- مفاتيح الـ APIs الحساسة ماينفعش تظهر في المتصفح
- المعالجة الثقيلة والتجميع والـ scoring أحسن تكون في الباك

### PDF Generation
استخدمنا توليد PDF داخل التطبيق لأن:

- المستخدم يقدر يحمل تقرير رسمي
- التقرير يبقى فيه تفاصيل فنية وإجرائية
- ينفع يتشارك في المناقشة أو مع فريق الأمن

## 4) تدفق البيانات

### فحص الباسورد
1. المستخدم يدخل الباسورد في `MoreToolsPage`
2. الواجهة تبعت الطلب للباك من خلال `securityAPI.analyzePassword()`
3. الباك يحلل:
   - الطول
   - وجود uppercase / lowercase
   - الأرقام
   - الرموز
   - الأنماط الشائعة
   - وجوده في بيانات breach
4. النتيجة ترجع للواجهة
5. الواجهة تعرض النتيجة وتسمح بتحميل PDF

### فحص الإيميل
1. المستخدم يدخل البريد
2. الواجهة تبعت الطلب للباك
3. الباك يعمل lookup على بيانات التسريب
4. النتيجة ترجع فيها:
   - عدد مرات الظهور
   - verified breaches
   - stealer logs
   - آخر breach
   - data classes exposed
5. النتيجة تظهر في UI ومعاها PDF

### فحص الويب
1. المستخدم يدخل URL من الداشبورد
2. الباك يشغل scanner
3. النتيجة ترجع فيها:
   - findings
   - severity counts
   - headers
   - parameters
   - forms
   - validation results
4. الواجهة تعرض summary + تفاصيل + Export PDF

## 5) أهم الحاجات اللي اتعملت

### تقارير PDF بقت أوضح
أضفنا شرح تفصيلي في التقارير يوضح:

- النتيجة معناها إيه
- مكان المشكلة فين
- ليه ده مهم
- نمنع المشكلة إزاي

### فحص الباسورد
بقى يرجع تفاصيل أغنى مثل:

- score
- strength
- entropy
- issues
- checks
- recommendations
- priority actions

### فحص الإيميل
بقى يوضح:

- breach count
- verified count
- stealer logs
- latest breach
- exposed data classes

### الداشبورد
بقى يعرض:

- كروت تحليل مختصرة
- View Report
- Export PDF
- تفاصيل أوضح عن findings

### إعدادات الأدمن
بقت بعض الإعدادات تحفظ مباشرة مع تغييرها، بدل ما تفضل فاضية أو غير متصلة.

## 6) ليه استخدمنا الملفات دي

### `src/services/api.js`
عشان نفصل كل نداءات الـ API في مكان واحد، وده يسهل الصيانة وتعديل endpoints لاحقًا.

### `breach_service.py`
عشان يبقى فيه مكان واحد مسؤول عن منطق تحليل الباسورد والإيميل بدل ما يتوزع في الراوتس.

### `DashboardPage.jsx`
عشان هو الصفحة الرئيسية لعرض نتائج الويب، والتقارير، والملخصات، والتصدير.

### `MoreToolsPage.jsx`
عشان الصفحة دي مخصصة لأدوات سريعة زي Password Checker و Email Checker مع PDF جاهز للتنزيل.

## 7) إزاي المشروع مفيد في المناقشة

تقدر تقول إن المشروع بيجمع بين:

- Security scanning
- Risk scoring
- Exposure intelligence
- Report generation
- Backend-backed sensitive checks

وده معناه إن التطبيق مش مجرد واجهة شكلها حلو، لكنه بيقدم workflow كامل:

Input -> Analysis -> Risk explanation -> PDF report -> Follow-up action

## 8) نقاط قوة معمارية

- فصل واضح بين الفرونت والباك
- عدم كشف المفاتيح الحساسة في المتصفح
- تقارير قابلة للتصدير
- بيانات منظمة بدل نصوص عشوائية
- قابلية التوسع لإضافة فحوصات جديدة

## 9) لو اتسألت عن سبب الاختيارات

- `React`: عشان UI ديناميكي وسهل التقسيم
- `Vite`: عشان سرعة التشغيل والبناء
- `Backend API`: عشان الحماية وفصل المنطق
- `PDF export`: عشان التقارير الرسمية
- `scoring`: عشان نحول النتائج إلى قرار واضح
- `cards + insights`: عشان العرض يبقى مفهوم وسهل القراءة

## 10) تشغيل المشروع

عادة التشغيل بيكون من:

- تشغيل الفرونت
- تشغيل الباك
- ثم فتح التطبيق من المتصفح المحلي

ولو في البيئة المحلية متوفرة، الأفضل تتأكد من:

- إعدادات `.env`
- اتصال MongoDB
- مفاتيح الـ APIs المطلوبة

## 11) خلاصة قصيرة
المشروع متقسم بطريقة تخدم الأمان وسهولة الاستخدام معًا:

- الواجهة تعرض النتائج بشكل واضح
- الباك يعمل التحليل الحقيقي
- الـ PDF يشرح المشكلة بطريقة تصلح للعرض والمناقشة
- الداشبورد يجمع كل شيء في مكان واحد
